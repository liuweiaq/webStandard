概述

本规范制定原则

方便代码的交流和维护；不影响编码的效率，不与大众习惯冲突；使代码更美观、阅读更方便； 使代码的逻辑更清晰、更易于理解。

适用范围

本规范主要适用于书写JavaScript代码的Web前端开发人员。

总体要求

  • 如无特殊说明，JS代码至少需要兼容chrome，firefox，ie9及以上版本浏览器；
  • 所有文件需统一采用UTF-8无BOM编码；
  • 避免每一行超过80个字符，当一条语句一行写不下时，需要考虑折行书写；
  • 代码组织应该清晰易懂，缩进单位为四个空格，不能使用tab键进行缩进；
  • JavaScript程序应独立保存在后缀名为.js的文件中，不能包含在HTML文件中；
  • 本文档规定所有JavaScript代码均须在严格模式(“use strict”)下运行，有关严格模式的要求请查阅相关文档；
  • 为方便后续代码编写及程序调试，需搭建以下基本软件环境：node.js(JavaScript运行环境，是其他环境运行的基础)、bower(包管理工具)、handlebars.js(JS模板引擎)、grunt(项目构建工具)、Nginx(HTTP和反向代理服务器)等，环境搭建教程见其他相关文档；
  • JS库引入：原则上仅能引用Jquery(JavaScript库)、RequireJS(AMD模块化管理工具)、Backbon(JavaScript的MVC框架)、BootStrap(响应式布局插件)四种JS库，若需要引入其他第三方库, 须与团队其他成员讨论决定。

命名规范

术语定义

  • Pascal大小写：
    将标识符的首字母和后面连接的每个单词的首字母都大写。可以对三字符或更多字符的标识符使用Pascal大小写。例如：BackColor。
  • Camel大小写：
    标识符的首字母小写，而每个后面连接的单词的首字母都大写。例如：backColor。
  • 全大写：
    标识符的所有字母都大写。例如：PI。

命名的基本原则

  • JavaScript命名的总体原则是：尽量与原生JavaScript中命名规则相同。
  • 名称均须使用具有实际含义的英文单词命名，不能使用JavaScript关键字，拒绝使用拼音等中式命名方法，产生歧义。
  • 允许使用公认英文缩写，如：getXML()。
  • 一般而言，方法、事件名称须是动词短语，其他命名按需求使用形容词或名词短语。
  • 除常量名称外，其他所有名称两个单词之间均不能用下划线连接。如：first_Name。

各成员命名方法

  • JavaScript文件名：
    JavaScript文件名需是名词短语，采用Camel命名法。如：countryModel.js。
  • 类名：
    类名都是名词短语，采用Pascal命名法。如：Person。
  • 方法(函数)名：
    方法采用动词短语，Camel方法进行命名。如：sayHi()。
  • 属性名：
    属性名是名词短语，采用Camel方法进行命名。如：name。

  • 变量名：
    如无特殊说明，变量名的命名规范符合字段、私有变量、方法参数等的命名要求。
    ①变量名均是名词短语，符合Camel命名规范。如：age。
    ②类的私有字段、私有属性、私有方法等成员的命名须在名称前加下划线(与公有成员进行区分)。如：_height;_getSize()。
    ③变量名可以加前缀(如下表所示)，表示特定含义。

    前缀名 含义 示例
    is 变量为布尔类型 isNumber
    str 变量为字符串类型 /
    arr 变量为数组 /
    obj 变量为对象 /
    num 变量为数字 /
    $ 变量为Jquery对象 $element
    _ 类的私有成员(字段、属性、方法等) _ height

    注：1)只有当需要特别说明其类型或用途的变量名才加前缀，所加前缀与后面名称组合的变量名要含义清晰；2)切记不要乱用前缀，如strName,numAge等有画蛇添足之嫌；3)前缀名仅限于以上几种，不能使用其他简写前缀名，以免产生歧义.
    ④在局部变量或参数名中，可以用下表中的简写代表特定的含义(如下表所示)。

    简写 含义
    o 表示Object
    e 表示Event
    el 表示Element
    elem 表示Element

    注：1)在代码中，上述四个名称不能再代表其他的含义；2)只有在局部变量和参数名中才能使用上述简写命名。

  • 常量名：
    常量使用全大写方法命名，如PI。注：常量名如果由几个单词组成，单词之间用下划线连接起来。

代码注释规范

注释原则

  • 注释形式必须统一：必须使用本文档规定的标点和结构的样式来构造注释。注释格式如下：①单行注释：”//…..”；②块注释：”/*………*/”。注：具体各代码段的注释样式，见本章注释样式。
  • 注释内容要简单、明了、含义准确，防止注释的多义性，错误的注释不但无益反而有害。注：文件头这里不做约定，有git版本控制，就知道文件的基本信息，而且一般开发人员对于文件头的维护不及时，容易造成错落。
  • 基本注释部分需符合JSDoc语法规范。JSDoc是JavaScript语言的一套代码注释规范，其主要命令名如下表所示：
    命令名 描述
    @param 指定参数名和说明来描述一个函数参数
    @return 函数的返回值
    @type 函数的返回类型
    @author 代码的作者
    @example 范例
    @description 说明
    @default 默认值
    @see 描述相关的资源
    注：这里我们只是列出了常用的标签，了解更多标签，可以登陆到官方网站查看(http://code.google.com/p/jsdoc-toolkit/wiki/TagReference)。

注释样式

本文档根据注释内容的不同，将代码注释分为基本注释和特殊必加注释两种。这里需要特别说明的一点是：代码中不必每一处都添加注释，但对需要重点说明的地方，一定要合理安排注释内容。

  • 基本注释：本文档中，基本注释包括类、方法、属性、主要变量等的注释。这部分内容是JS代码的基本组成部分，这部分内容需要用块注释格式。①类的注释： 
    /*** @description 一个Person类* @param {String} userName 姓名* @example new Parent(“张三”,15);**/
    ②方法的注释：方法的注释需包含的内容有： 
    /*** @description 弹出say内容* @pparam {String} content 内容* @return {String} json格式数据**/
    ③属性和字段的注释：属性和字段的注释需包含的内容有： 
    /*** @description 弹出say内容* @description {Sting} 姓名**/
  • 特殊必加注释：除了基本注释外，代码中有以下情形之一者，也需要添加注释：①典型算法处；②代码不明晰处；③代码修改处需要加上修改标识的地方；④循环和逻辑分支组成的代码中；⑤为他人提供的接口。这部分注释只需用行注释格式，简洁准确地标明注释内容即可。

JavaScript及库使用规范

原生 JavaScript使用约定

  • 每一条JavaScript语句都要以分号结束：严格按照语法规范书写代码，保证代码在压缩后不会出错。
  • 变量定义：在类、方法体顶端定义所有变量，var语句应该为类或方法体内的第一条语句。同时定义多个变量时，省略var关键字，用逗号代替。
    Bad Code Good Code
    var someItem = 'some string'; var someItem = 'some string';
    var anotherItem = 'another string'; var anotherItem = 'another string';
    var oneMoreItem = 'one more string'; var oneMoreItem = 'one more string';
  • 使用”===”代替”==”：JavaScript 使用2种不同的等值运算符：===|!== 和 ==|!=，后者附带强制类型转换，为了避免出现比较时的错误，统一使用前者作为JS代码中的等值运算符。
  • 使用 join() 来拼接字符串：字符串的拼接在开发中会经常遇到，用“+=”的方式来拼接字符串的效率非常的低。因此在大量拼接时，规定统一使用数组的“.join()”方法来拼接字符串。如： 
    var array = [];for(var i = 0; i < 10000; i++){   array[i] = "<input type='button' value='a'>";}document.getElementById("one").innerHTML = array.join("");
  • 使用{}代替 new Ojbect()，使用[]代替 new Array()对象和数组分别有上述两种创建方式，本文档规定统一使用{}和[]来创建对象或数组。
  • 尽量避免全局变量将多个全局变量定义在一个命名空间下时，将增大变量污染的可能性。
    Bad Code Good Code
    var name = 'Jeffrey';
    var lastName = 'Way';
    function doSomething() {...}
    console.log(name);
    //Jeffrey或window.name    		 var DudeNameSpace = {
    name : 'Jeffrey',
    lastName : 'Way',
    doSomething : function() {...}
    }
    console.log(DudeNameSpace.name);
    // Jeffrey
  • 选择短路求值：短路求值是一个从左到右求值的表达式，用&&（逻辑与）或 || （逻辑或）操作符。
    Bad Code Good Code
    function initVar($myVar) {
    if(!$myVar) {
    $myVar = $('#selector');
    }
    }    		 function initVar($myVar) {
    $myVar = $myVar || $('#selector');
    lastName : 'Way',
    }

RequireJS使用约定

RequireJS是一个遵守AMD规范的客户端模块管理工具库，其基本思想是：通过define方法将代码定义为模块；通过require方法实现代码的模块加载。

  • 定义模块每一个模块均须定义在一个单独的JS文件里(注：本节中，如无特别说明，模块两字均表示这种)，根据模块依赖情况和用途的不同，定义形式也有所区别：①数据模块这类模块不需要依赖其他任何模块，一般用于存放数据。如： 
    define({    addUrl:"/Sync10/basicdata/countrys/addCountry",    modUrl:"/Sync10/basicdata/countrys/updateCountry",    getUrl:"/Sync10/basicdata/countrys/getCountry"});
    ②代码模块被定义的模块需依赖其他模块。除数据模块外，其他模块都使用这种方法定义。如： 
    define(['module1','module2'],function(m1,m2){    ......});
  • 调用模块模块调用根据目的的不同，也有两种调用方式：①执行入口mian.js中的程序执行入口处，用require方法调用模块，其参数与define方法类似。如： 
    require(['requireConfig'],function(requireConfig){    ......});
    ②依赖注入除程序执行入口外，其他模块均通过依赖注入的方式被新模块调用。调用方式采用本节代码模块中的规范。
  • 配置require.jsrequire方法本身也是一个对象，它带有一个config方法，用来配置require.js运行参数。config配置方法请参阅相关文档，示例如下： 
    requirejs.config({    paths:{        "oso.lib":"/Sync10-ui/lib"    },    shim: {        'jquery.dynamicTable': ['jquery']    }});
  • 模块化总体规范除了满足上述模块定义、模块调用以及配置等基本规范外，每一个功能模块(注：功能模块指的是能完成一定功能的程序集，在Backbone框架中，一个功能模块拥有一个SPA页)也需要满足一定要求：①每一个功能模块最多只能有一个main.js(程序入口)、一个requireConfig.js(RequireJS的配置文件，可选)、一个config.js(功能模块的配置信息，参考数据模块规范)文件。②以上三个文件名称不能随意更改。③在main.js中，一般先调用全域RequireJS配置文件。如： 
    require(["/requireConfig"], function() { //全域RequireJS配置注入    require(["<本模块目录> / requireConfig"], function() {     //本模块RequireJS配置注入，如果本模块没有requireConfig这一步可省略    },        require(["jquery", "models/admin"], function($, AdminModel) {             ......        });    });});

Backbone使用约定

  • 视图是数据无关的数据属于模型或集合，不能在视图或者DOM中存储数据，须做到数据层和视图层的关注点分离。
  • DOM事件只改变model当一个DOM事件触发时，不要让它改变view本身，而是去改变这个model。
  • DOM只有在model发生改变时才改变只有当model发生改变时，才能相应地改变其对应的DOM(或view)。
  • 绑定的东西必须解绑当视图从DOM上移除时，使用”remove”方法，视图必须从所绑定的事件上解绑。
  • 视图的作用域视图不要操作除它本身以外的DOM。当你需要操作另一个视图时，可以触发一个事件，让其自身去完成。
  • 组件封装组件封装时，用暴露view的方式提供接口给用户。如果model为某一view私有，就无需单独写一个model的JS文件，而是把model与view定义在一个命名空间下，封装起来。
  • Backbone中路由(router)的处理在程序中需要根据需求，合理设置Backbone路由，满足客户要求。本文档规定，使用Hash规则设定路由。
